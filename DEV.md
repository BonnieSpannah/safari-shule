# Safari Shule — Developer Reference

## Development modes

### Recommended: Native API + Docker infra (fastest iteration)

Docker runs **only infrastructure** (Postgres, Redis, Mailhog). The API and web run natively on your machine with hot-reload. Code changes go live in **~2 seconds** — no image rebuild ever needed.

```bash
# Terminal 1 — one-time infra boot (keeps running)
make infra

# Terminal 2 — NestJS watch mode (~2s reload on save)
make api-dev

# Terminal 3 — Vite HMR (<1s reload on save)
make web-dev
```

### Full Docker stack (CI-parity, slower iteration)

Every API source change requires a rebuild (~3-4 min). Use this only to validate the production image locally or in CI.

```bash
make up                                                        # build + start everything
docker compose -f infra/docker-compose.yml --env-file .env \
  up -d --force-recreate api                                   # redeploy API after changes
```

## Scenario guide

### Fresh clone / first-time setup

```bash
node --version           # must be 20.11.0 (see .nvmrc)
nvm use                  # switch to correct Node
pnpm install             # install all workspace deps

cp .env.example .env     # create local env (fill in any blanks)

make infra               # start postgres + redis + mailhog
make db-migrate          # apply all migrations
make db-generate         # regenerate Prisma client
make db-seed-local       # seed hillcrest demo tenant + fixtures

make api-dev             # NestJS hot-reload
make web-dev             # Vite HMR
```

Demo credentials after seed (see seed output for generated secrets):
- Tenant: `hillcrest`
- Admin: `admin@hillcrest.ac.ke` / `Demo!Password1`

### Adding a database migration

1. Edit `apps/api/prisma/schema.prisma`
2. Run interactively (creates the SQL file + applies it):
   ```bash
   make db-migrate-new NAME=add_something_meaningful
   ```
3. Commit the generated migration file in `apps/api/prisma/migrations/`
4. Prisma client is automatically regenerated by `migrate dev`

> **Never** hand-edit the migration SQL unless you know exactly what you're doing.
> If `migrate dev` wants to reset (schema drift), see [Resolving drift](#resolving-schema-drift) below.

---

### Pulling new code that contains a migration

```bash
git pull
pnpm install             # only if package.json changed
make db-migrate          # apply new migration(s) — safe, never destructive
make db-generate         # regenerate Prisma client with new types
```

The API restarts automatically via watch mode — no further action needed.

---

### After editing `schema.prisma` without creating a migration (e.g. adding a comment)

```bash
make db-generate         # just regenerates the Prisma client
```

### Resolving schema drift

Drift occurs when the DB state doesn't match the migration history (e.g. a column was added directly via SQL). `prisma migrate dev` will warn and may offer to reset.

**Do NOT reset if you have dev data you care about.**

Instead:
1. Write the SQL manually (e.g. `ALTER TABLE ... ADD COLUMN ...`)
2. Apply it directly:
   ```bash
   docker exec safari-postgres psql -U safari -d safari_shule -c 'ALTER TABLE ...'
   ```
3. Create the migration file:
   ```
   apps/api/prisma/migrations/NNNN_migration_name/migration.sql
   ```
4. Mark it as applied:
   ```bash
   pnpm --filter @safari-shule/api exec prisma migrate resolve --applied NNNN_migration_name
   ```
5. Regenerate the client:
   ```bash
   make db-generate
   ```

### Re-seeding without dropping the DB

The seed script is idempotent for the `hillcrest` tenant — it uses `upsert` and skips existing records.

```bash
make db-seed-local
```

---

### Destructive reset (wipes ALL dev data)

```bash
make reset               # drops DB, re-migrates, re-seeds (runs inside Docker container)
```

Or natively:
```bash
pnpm --filter @safari-shule/api exec prisma migrate reset --force
make db-seed-local
```

## How the API finds its `.env`

`ConfigModule` is configured with `envFilePath: ['../../.env', '.env']`:
- **Native dev**: reads the root `../../.env` (relative to `apps/api/`) — connection strings point to `localhost`
- **Docker**: env vars are injected directly by compose `environment:` block, overriding the `.env` values with Docker service hostnames (`pgbouncer`, `redis`)

This means the root `.env` always uses `localhost` addresses and Docker compose always wins inside containers — no separate env files needed.

## When to rebuild the Docker API image

Only needed when deploying to **staging or production**, or to validate the production image locally:

```bash
DOCKER_BUILDKIT=0 docker build --no-cache -f infra/Dockerfile.api -t safari-shule-api:latest .
docker compose -f infra/docker-compose.yml --env-file .env up -d --force-recreate api
make migrate             # apply any pending migrations inside the container
```
